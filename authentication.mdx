# Authentication

The Cold Navigator API uses API keys to authenticate requests. Every request must include a valid API key in the `x-api-key` header.

## Creating an API key

1. Log in to [Cold Navigator](https://app.coldnavigator.com)
2. Go to **Settings > API Keys**
3. Click **Create API Key**
4. Enter a descriptive name (e.g., "CRM Sync", "Analytics Dashboard")
5. Select the scopes your integration needs
6. Click **Create**

<Warning>
  Your API key is displayed only once at creation time. Copy it immediately and store it in a secure location such as a secrets manager or environment variable. If you lose it, you will need to create a new key.
</Warning>

## Using your API key

Include the API key in the `x-api-key` header with every request:

<CodeGroup>
  ```bash cURL
  curl -X GET https://app.coldnavigator.com/api/v1/public/leads \
    -H "x-api-key: cn_live_a1b2c3d4e5f6..."
  ```

  ```javascript JavaScript
  const response = await fetch("https://app.coldnavigator.com/api/v1/public/leads", {
    headers: {
      "x-api-key": "cn_live_a1b2c3d4e5f6..."
    }
  });
  ```
</CodeGroup>

<Note>
  Always use HTTPS. Requests over plain HTTP will be rejected.
</Note>

## Scopes

API keys are scoped to limit what actions they can perform. Assign only the scopes your integration needs — this follows the principle of least privilege.

| Scope | Description |
|-------|-------------|
| `read:leads` | Read lead data and lead groups |
| `write:leads` | Create, update, delete leads, manage tags and groups |
| `read:campaigns` | Read campaign data and enrolled leads |
| `write:campaigns` | Start, pause, resume campaigns, enroll/unenroll leads |
| `read:conversations` | Read conversations and messages |
| `read:accounts` | Read LinkedIn account information |
| `read:analytics` | Read analytics and reports |
| `write:linkedin_actions` | Send connection requests, messages, InMails, view profiles |
| `webhooks:manage` | Create, update, delete webhooks |
| `*` | Full access (all scopes) |

<Warning>
  The `*` (wildcard) scope grants unrestricted access. Only use it for trusted, internal integrations. For third-party or client-facing integrations, always use the narrowest scopes possible.
</Warning>

If a request requires a scope that the API key does not have, the API returns a `403 Forbidden` error:

```json
{
  "error": {
    "message": "API key missing required scope: write:leads",
    "code": "INSUFFICIENT_SCOPE"
  }
}
```

## Rate limits

Rate limits vary by plan. Limits are applied per API key.

| Plan | Requests per minute | Requests per day |
|------|--------------------:|-----------------:|
| Free Trial | 30 | 1,000 |
| Single | 60 | 5,000 |
| Team | 120 | 20,000 |
| Growth | 300 | 100,000 |

<Note>Free plan users do not have API access. Upgrade your plan to use the API.</Note>

### Rate limit headers

Every response includes headers that tell you your current rate limit status:

| Header | Description |
|--------|-------------|
| `RateLimit-Limit` | Maximum requests allowed in the current window |
| `RateLimit-Remaining` | Requests remaining in the current window |
| `RateLimit-Reset` | Seconds remaining until the current window resets |

Example response headers:

```
RateLimit-Limit: 60
RateLimit-Remaining: 42
RateLimit-Reset: 58
```

<Note>
  Cold Navigator uses the [IETF RateLimit header fields](https://datatracker.ietf.org/doc/draft-ietf-httpapi-ratelimit-headers/) standard. Note that `RateLimit-Reset` returns the number of **seconds remaining** in the current window, not a Unix timestamp.
</Note>

### Handling rate limits

When you exceed the rate limit, the API returns a `429 Too Many Requests` response:

```json
{
  "error": {
    "message": "Rate limit exceeded. Please slow down your requests.",
    "code": "RATE_LIMIT_EXCEEDED"
  }
}
```

Best practices for handling rate limits:

- Check `RateLimit-Remaining` before making requests
- Use the `RateLimit-Reset` header to schedule retries
- Implement exponential backoff when you receive a `429` response
- Cache responses when possible to reduce the number of API calls

## Error codes

The API uses standard HTTP status codes. Here are the authentication and authorization errors you may encounter:

| Status | Code | Description |
|--------|------|-------------|
| `401 Unauthorized` | `INVALID_API_KEY` | The API key is missing, invalid, or has been revoked. |
| `403 Forbidden` | `INSUFFICIENT_SCOPE` | The API key does not have the required scope for this endpoint. |
| `403 Forbidden` | `PLAN_UPGRADE_REQUIRED` | API access is not available on the Free plan. |
| `429 Too Many Requests` | `RATE_LIMIT_EXCEEDED` | You have exceeded your plan's per-minute rate limit. |
| `429 Too Many Requests` | `DAILY_RATE_LIMIT_EXCEEDED` | You have exceeded your plan's daily rate limit. |

### 401 — Invalid API key

```json
{
  "error": {
    "message": "Invalid or expired authentication token",
    "code": "INVALID_API_KEY"
  }
}
```

Common causes:
- Typo in the API key
- The key has been revoked from Settings
- Using a test key against the production API or vice versa

### 403 — Insufficient scope

```json
{
  "error": {
    "message": "API key missing required scope: write:leads",
    "code": "INSUFFICIENT_SCOPE"
  }
}
```

Solution: Create a new API key with the required scope, or update the existing key's permissions from **Settings > API Keys**.

### 403 — Free plan blocked

```json
{
  "error": {
    "message": "API access is not available on the Free plan. Please upgrade to use the API.",
    "code": "PLAN_UPGRADE_REQUIRED"
  }
}
```

Solution: Upgrade to a paid plan (Free Trial, Single, Team, or Growth) to enable API access.

### 429 — Per-minute rate limit

```json
{
  "error": {
    "message": "Rate limit exceeded. Please slow down your requests.",
    "code": "RATE_LIMIT_EXCEEDED"
  }
}
```

Solution: Wait until the `RateLimit-Reset` countdown reaches 0, then retry.

### 429 — Daily rate limit

```json
{
  "error": {
    "message": "Daily rate limit exceeded. Please try again tomorrow.",
    "code": "DAILY_RATE_LIMIT_EXCEEDED"
  }
}
```

Solution: You have exhausted your plan's daily quota. Wait until the next day or upgrade to a higher plan.

## Response format

All API responses follow a consistent format.

### Success responses

```json
{
  "data": {
    "id": "f7e6d5c4-b3a2-1098-7654-321fedcba098",
    "first_name": "Jane",
    "last_name": "Smith"
  },
  "message": "Success"
}
```

The `data` field contains the response payload. The `message` field is optional and provides a human-readable description.

### Error responses

```json
{
  "error": {
    "message": "Human-readable error message",
    "code": "ERROR_CODE",
    "details": [
      {
        "field": "linkedin_url",
        "message": "Must be a valid LinkedIn profile URL"
      }
    ]
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `error.message` | string | A human-readable description of the error |
| `error.code` | string | A machine-readable error code (optional) |
| `error.details` | array | Field-level validation errors (optional, present on `400` responses) |

## Security best practices

<CardGroup cols={2}>
  <Card title="Use environment variables" icon="lock">
    Never hard-code API keys in source code. Store them in environment variables or a secrets manager.
  </Card>
  <Card title="Rotate keys regularly" icon="rotate">
    Create new keys periodically and revoke old ones. This limits the impact if a key is compromised.
  </Card>
  <Card title="Use minimal scopes" icon="shield-check">
    Only grant the scopes your integration actually needs. Avoid using the wildcard scope in production.
  </Card>
  <Card title="Monitor usage" icon="chart-bar">
    Review your API key activity in Settings to detect unexpected usage patterns.
  </Card>
</CardGroup>
